# 公司官网图片与转化链路重构设计文档

- 日期：2026-05-14
- 项目路径：`/data/work/web/mycompanysite`
- 关联文档：`docs/superpowers/specs/2026-05-13-company-website-design.md`
- 本轮目标：在现有中文官网基础上，完成图片资产重整、视频修复、品牌图标修正、联系表单通知链路补强，以及提交成功后的微信二维码继续沟通闭环。

## 1. 本轮背景

当前站点已经完成基础企业官网落地，但仍存在以下影响上线质量的问题：

1. 首页核心业务卡片仍有未替换或不匹配的图片
2. 首页短片无法正常播放
3. 导航栏品牌图标不正确
4. 典型场景案例区图片错配，且部分素材质量不稳定
5. 联系表单提交后，站点没有明确告诉站方如何查看用户联系方式
6. 全站多处内容图存在重复使用，削弱专业感
7. 解决方案页最后一段仍使用场景图，不能准确表达“方案服务”这一方法论性质
8. 联系表单提交成功后，缺少进一步引导用户继续沟通的轻量闭环

本轮改造的核心不是单点修补，而是做一轮围绕“可信展示 + 真实转化”的体验重构。

## 2. 已确认决策

### 2.1 整体方案

采用“体验重构型”方案，在不大规模改写页面结构的前提下，完成：

- 图片资产重新分配
- 视频链路修复
- 品牌图标修正
- 联系表单通知闭环建立
- 二维码成功态交互补足
- 与上述内容直接相关的小范围版式微调

### 2.2 联系表单通知方案

用户已确认：

- 收件邮箱：`15340530127@163.com`
- 发件方式：使用已授权的 Google Workspace 能力发送通知邮件
- 成功态形式：居中弹窗 modal
- 二维码素材：`/data/work/web/QR_code/联系二维码.jpg`

### 2.3 补图策略

采用“混合方案”：

1. 先尽量使用项目现有素材与 `PPT/Word` 提取素材
2. 素材不足时，补充少量网络实景行业图
3. 网络补图必须以实景、行业相关、风格统一为原则

## 3. 目标与边界

### 3.1 本轮目标

本轮交付需要满足以下目标：

- 首页四个核心业务卡片全部替换为正确且互不重复的图片
- 首页视频在本地部署态和正式构建产物中可正常播放
- 导航栏品牌图标替换为正确素材，并在现有主题下保持清晰
- 典型场景案例区三张图片与各自场景一一对应
- 全站避免内容图片区重复使用同一张图片
- 解决方案页最后一段使用黑色 ICON 大图替代场景照片
- 联系表单提交成功后：
  - 站方能通过邮件直接查看联系方式
  - 用户能在页面内看到提交成功反馈
  - 用户能立即扫码继续沟通

### 3.2 非目标

本轮不包含以下范围：

- 全站文案大改写
- 新增完整 CRM 后台
- 新增用户登录或线索管理后台
- 改写整个站点的组件体系
- 无关页面的全面视觉翻新

## 4. 设计原则

### 4.1 图片唯一性优先

品牌功能素材可以复用，但内容图片区原则上不复用：

- 允许复用：品牌 logo、二维码、站点图标
- 不允许复用：首页业务卡片图、首页案例图、解决方案场景图、关于页主展示图

### 4.2 场景语义优先于装饰感

图片选择以“看图就能理解业务方向”为优先，不使用仅提供科技氛围但缺少业务含义的抽象图作为主要内容图。

### 4.3 成功态必须形成闭环

表单提交成功后，不只显示一句提示，而是同时完成两件事：

1. 让站方可以收到联系人信息
2. 让用户能立即进入下一条沟通通道

### 4.4 小范围重构，不做整站重做

允许围绕首页、联系页、解决方案页做局部版式调整，但不扩展为整站重构项目。

## 5. 图片与素材重构方案

### 5.1 首页核心业务卡片

首页四个核心业务卡片需要分别映射到独立图片，不再与其他模块重复使用：

1. 无人化零件供应
2. 核心部件供应
3. 模块化通用底盘
4. 方案服务

#### 设计要求

- 每张图都要与标题语义一致
- 四张图之间视觉风格保持统一
- 不再复用首页案例区图片
- 不再复用解决方案页场景图

#### 素材来源优先级

1. 项目现有已导入图片
2. `PPT/Word` 提取图
3. 少量外部实景行业图

### 5.2 典型场景案例区

案例区重建为三类更明确的场景表达：

1. 丘陵山地农业
2. 园区巡检
3. 公共服务 / 复合任务

#### 设计要求

- 三张图与三个案例标题一一对应
- 图像内容优先选用实景图，而不是抽象插图
- 不与首页业务卡片、解决方案图片区重复
- 若现有标题与图像语义差距过大，允许微调标题或摘要以匹配新图

### 5.3 解决方案页图片区

解决方案页前三段维持“场景实景图”表达，第四段“从需求诊断到交付验证的方案服务”改为黑色 ICON 大图。

#### 设计目的

通过视觉语言区分：

- 前三段：真实业务场景
- 第四段：方法论 / 服务流程 / 平台化能力

#### 设计要求

- ICON 图必须为黑色主视觉
- 视觉上应明显区别于照片素材
- 大图本身需要具备足够信息密度，避免显得空洞

### 5.4 导航品牌图标

导航品牌图标需要替换为正确品牌图标，并检查：

- 当前使用资源是否错误
- 深浅背景上的可视性
- 尺寸、留白、反色或滤镜是否导致图标失真

本轮只修正品牌图标，不把它与内容图片区混用。

### 5.5 全站图片去重映射

需要建立一份明确的图片分配表，至少覆盖：

- 首页核心业务卡片
- 首页典型场景案例
- 解决方案页 section 图片
- 关于页主视觉或优势图片区

目的不是单纯“找几张不一样的图”，而是让后续维护者能清楚知道每张图负责哪个业务表达。

## 6. 视频修复方案

### 6.1 问题定义

首页视频当前存在“无法播放”问题。对外表现上，无论根因是文件缺失、路径不对、构建未带上还是浏览器兼容性问题，都应视为同一个交付缺陷：视频模块未形成真实可用体验。

### 6.2 修复目标

视频修复后的标准不是“代码引用了视频”，而是：

- 首页在本地部署态下可正常加载视频
- 构建后的部署产物可正常访问视频资源
- 浏览器可见 controls，用户可主动播放
- 视频资源至少具备稳定的首帧或可加载元数据

### 6.3 处理策略

按以下优先顺序处理：

1. 检查当前视频文件是否真实存在且路径正确
2. 检查构建流程是否把视频文件带入最终产物
3. 检查浏览器实际请求路径与响应状态
4. 若源视频本身不适合作为网页播放素材，则替换为稳定可播放的视频文件

本轮不要求增加复杂视频播放器能力，只要求修复站内原生视频播放链路。

## 7. 联系表单与通知闭环设计

### 7.1 现有链路

现有联系表单前端通过 `/api/contact` 提交用户信息。本轮在此基础上补足“接收方如何查看联系方式”与“用户提交后如何继续沟通”两部分能力。

### 7.2 新链路目标

用户提交表单后，需要形成以下闭环：

1. 后端成功接收数据
2. 发送通知邮件到 `15340530127@163.com`
3. 前端显示提交成功反馈
4. 当前页中央弹出微信二维码 modal
5. 用户可扫码继续沟通

### 7.3 邮件通知内容

邮件至少包含以下字段：

- 提交时间
- 姓名
- 公司名称
- 联系方式
- 需求类型
- 需求描述
- 来源页面（如可获取）

通知邮件的目标是让收件人看到邮件即可直接拿到联系方式，不需要再依赖浏览器控制台、日志文件或数据库后台。

### 7.4 发信方式

本轮默认采用用户已授权的 Google Workspace 能力发信。

#### 设计要求

- 发信逻辑由后端触发，不由前端直连第三方服务
- 邮件发送失败时，后端应返回可判断状态
- 前端不能在后端未确认接收成功时就直接展示成功弹窗

### 7.5 成功态交互

成功态确定采用“居中弹窗 modal”。

#### modal 内容

- 成功标题
- 简短说明：已收到需求，可扫码继续沟通
- 微信二维码图片
- 关闭按钮
- 保留用户停留在联系页的能力

#### 交互原则

- 不跳整页成功页
- 不在提交前打断用户
- 不把扫码作为提交前置条件
- 只在后端确认成功后弹出

### 7.6 二维码素材使用方式

二维码图片来源：`/data/work/web/QR_code/联系二维码.jpg`

展示要求：

- 在 modal 中作为主视觉显示
- 配套简短引导文案
- 移动端也能完整显示，不被裁切

## 8. 页面结构与体验重构范围

### 8.1 首页

首页将被作为一个整体体验来校正，而不是分别零碎修补。重点联动以下三个区块：

- 核心业务卡片
- 视频展示区
- 典型场景案例区

目标是解决当前“有的图像像占位、有的图像像实景、视频又单独失效”的割裂感。

### 8.2 导航栏

导航品牌区本轮同时检查：

- 正确图标文件
- 当前尺寸与间距
- 主题背景上的可读性
- 是否存在不必要滤镜导致显示异常

### 8.3 联系页

联系页不重做整个表单布局，但增加成功反馈层：

- 提交成功后弹出二维码 modal
- modal 内部文案保持商务官网语气
- 弹窗关闭后仍停留在当前页

### 8.4 小范围版式微调

若换图后某些区块出现明显失衡，允许做与本轮目标直接相关的小范围版式调整，例如：

- 图片容器比例修正
- 文案与图片的上下间距微调
- 成功态弹窗的移动端适配

但不扩展为无关页面的整页重做。

## 9. 风险与控制策略

### 9.1 图片来源风险

风险：`PPT/Word` 中可直接使用的图片不足或质量不稳定。

控制策略：

- 先做素材盘点
- 优先用现有素材
- 仅在必要时补少量外部实景图
- 外补图片必须围绕无人化装备、农业、巡检、园区、公服场景

### 9.2 邮件发信链路风险

风险：当前项目里未必已有可直接复用的 Google Workspace 发信模式。

控制策略：

- 先确认 `/api/contact` 现状
- 明确后端发信触发点
- 若当前运行环境需要配置凭据，则把凭据要求写清楚，不伪装成已完成联调

### 9.3 成功态误判风险

风险：前端在后端未确认成功时提前弹出二维码，导致误导用户。

控制策略：

- 以后端成功响应作为唯一弹窗触发条件
- 邮件发送失败与表单接收失败要区分状态

### 9.4 范围扩散风险

风险：在修图片与成功态时顺手扩大到整站重构。

控制策略：

- 仅修改与你这 8 项需求直接相关的页面与组件
- 无关区块不做额外改造

## 11. 验收标准

### 11.1 图片与品牌

- 首页四张核心业务图均已替换，且互不重复
- 首页案例区三张图与场景一一对应，且互不重复
- 解决方案页最后一段已替换为黑色 ICON 大图
- 导航品牌图标显示正确
- 全站无明显错图、乱图、重复图

### 11.2 视频

- 首页视频在本地部署态下可加载并可播放
- 视频资源请求返回成功，不出现 404 或空白播放器

### 11.3 表单与通知

- 联系表单可正常提交
- 提交成功后页面弹出居中二维码 modal
- 成功态文案清晰
- 收件邮箱 `15340530127@163.com` 能收到包含联系方式的通知邮件，或在当前环境受限时有明确可验证的阻塞说明

### 11.4 体验一致性

- 联系页成功态在桌面端与移动端都可正常使用
- 首页图片、视频、案例区风格更加统一
- 改动不破坏现有主导航与关键页面点击链路

## 12. 实施顺序建议

1. 盘点并分配图片素材，建立去重映射
2. 替换首页四张业务卡片与案例区图片
3. 修正导航品牌图标
4. 替换解决方案页最后一段为黑色 ICON 大图
5. 修复首页视频播放链路
6. 审查 `/api/contact` 当前实现与数据流向
7. 接入 Google Workspace 邮件通知
8. 增加提交成功后的二维码 modal
9. 做本地部署态点击验收与关键链路验证

## 13. 结论

本轮改造不是简单换几张图，而是围绕“展示可信度”和“线索转化闭环”做一轮有边界的体验重构。交付完成后，站点需要在视觉素材、品牌识别、视频可信度与联系转化路径上都更接近真实可上线的企业官网状态。
